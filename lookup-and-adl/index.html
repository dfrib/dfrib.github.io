WIP Internal linkage and a curious case of ADL

cplusplus
Tags// cpp, lookup, linkage,

Is there any scenario where we should prefer using static instead of an unnamed namespace to give a name internal linkage?

The other day I was in a situation where I wanted to hide a function name from other translation units (TU) than its own (internal linkage), but I also needed it to be found via Argument-dependent lookup (ADL) from a scope within its own TU that could not find it via (un-)qualified name lookup. For namespace-scope functions I usually rely on unnamed namespaces for internal linkage but in this case I had to rely on the static class storage specifier, as ADL does not, opposite to lexical name lookup, continue beyond its own namespace. Details follow below.

A recap of linkage

A name in C++, as (re-)introduced by a declaration, can have several properties, one of which is its linkage, which is either given implicitely by context and scope, or explicitly specified by the storage class specifiers of the name’s declaration. The linkage of a name is one of the following:

  • no linkage
  • internal linkage
  • external linkage

No linkage is not of particular interest for the context of this post, but basically refers to names which can only be referred to from within the scope in which the name has been declared; for example a variable name declared at block scope without the extern storage class specifier. We’ll focus on the two latter two for the remainder of this post.

Internal linkage

Internal linkage, as compared to no linkage, expands the scope of reference of a name to all scopes in the TU in which it has been declared. All names declared within an unnamed namespace have internal linkage, but e.g. variable and function names can be given internal linkage by be declaring them using the static storage class identifier.

So what does it look like we try access a name with internal linkage from another TU than the one in which is has been defined? In the following snippets the function name a(), declared (and defined) in an unnamed namespace in the namespace foo, has internal linkage to the TU associated with the foo.cpp source file.

// foo.h
#pragma once

// foo.cpp
#include "foo.h"
#include <iostream>

namespace foo {
namespace {

void a() { std::cout << "internal linkage\n"; }

}  // namespace
}  // namespace foo

foo::a() can thus not be referred to from the scope different TU’s, say the TU associated with the main.cpp source file in the snippet below, even if the name is (re-)declared in main.cpp.

// main.cpp
#include "foo.h"

namespace foo {

void a();

}  // namespace foo

int main() {
    foo::a();
    return 0;
}

From my experience undefined reference errors can be a bit elusive for inexperienced developers, particularly in large opaque code bases, but knowing about linkage as well as difference between declarations and definitions can go a long way to mitigate this, and my first hint when junior colleagues are struggling with these kind of errors usually go along the lines of:

– Has the name for which you get an undefined reference error been defined?
– What is the linkage of the name for which you get the undefined reference error?

[Rant] Avoiding special cases in order to avoid others having to rely on knowing them by heart

A common special case for implicit internal linkage applies for a variable name with namespace scope if it is:

  • a non-template (C++17: non-inline) variable of non-volatile const-qualified type that is neither explicitly declared extern nor previously declared to have external linkage,

even if it is not explicitly declared within an unnamed namespace nor declared with static. This is not a special case from some dark corner of standard, but it can still be used to argue about an general approach when working with any kind of special cases: to prompt fellow well-versed developers to avoid writing code that relies on others knowing the special cases by heart. If a special case can be replaced with a more general alternative approach, I would recommend to always do so. E.g. for this particular example, always declare variable names within an unnamed namespace if the intent is for the variable name to have internal linkage, even if they may fall under a special case and have, implicitly, internal linkage. A risk with relying on special cases is that they can lead to future mishaps when another developer comes along and follows the existing pattern of the source code without knowing about the details. E.g. in this particular example declaring another name, which does not fall under the same special case, in the same scope, mistakenly thinking one follows the practice on the pre-existing (“internal linkage intended”) source code.

To wrap up this sub-section with a contrived example; replacing this scenario:

// bar.cpp, v.1,
// by someone well-versed in the standard
namespace bar {

const int a = 42;  // internal linkage

// ... a is used and intended to only ever be used in the
//     scope of this source file / TU.

}  // namespace bar

// bar.cpp, v.2,
// modified by a somewhat novice C++ developer (and sloppy peer review)
namespace bar {

const int a = 42;  // internal linkage
int b       = 84;  // external linkage

// ... a and b is used and intended to only ever be used
//     in the scope of this source file / TU.

// b can be referred to, and written to, from other TU's!

}  // namespace bar

in favour of the following one:

// bar.cpp, v.1,
// by someone well-versed in the standard with a mindset of future-proofing
namespace bar {

namespace {

const int a = 42;  // internal linkage

}  // namespace

// ... a is used and intended to only ever be used in the
//     scope of this source file / TU.

}  // namespace bar

// bar.cpp, v.2,
// modified by a somewhat novice C++ developer
namespace bar {

namespace {

const int a = 42;  // internal linkage
int b       = 84;  // internal linkage

}  // namespace

// ... a and b is used and intended to only ever be used
//     in the scope of this source file / TU.

}  // namespace bar

External linkage

External linkage means that a name can be referred to from scopes in other TUs than the one in which it has been declared. This applies for e.g. functions and variables (barring the special case above) declared at namespace scope, given that they do not reside within an unnamed namespace nor have been declared with the static class storage specifier.

We modify the internal linkage example above such that, in the following snippets, the function name a(), declared (and defined) at namespace scope in the namespace foo, has external linkage.

// foo.h
#pragma once

// foo.cpp
#include "foo.h"
#include <iostream>

namespace foo {

void a() { std::cout << "external linkage\n"; }

}  // namespace foo

foo::a() can thus not be referred to from the scope different TU’s, say the TU associated with the main.cpp source file in the snippet below, even if the name is (re-)declared in main.cpp.

// main.cpp
#include "foo.h"

namespace foo {
void a();
}  // namespace foo

int main() {
    foo::a();
    return 0;
}

external linkage

One may also explicitly make use of the extern class storage specifier to declare a variable name to have external linkage, which can be used to e.g. declare the special case above, a non-volatile, non-template (C++17: non-inline) const-qualified variable, to have external linkage. As for the possible conflicts between explicit internal and external linkage mechanismsâconflicts we should not hope to see in well-written codeânote that an unnamed namespace takes linkage precedence over extern, whereas attempting to declare a variable name with both extern and static is an error.

// bar.cpp
namespace bar {

int a              = 0;  // external linkage
extern int b       = 0;  // external linkage
const int c        = 0;  // internal linkage
extern const int d = 0;  // external linkage

namespace {

int e              = 0;  // internal linkage
extern int f       = 0;  // internal linkage
const int g        = 0;  // internal linkage
extern const int h = 0;  // internal linkage

} // namespace

// [gcc] error: conflicting specifiers in declaration ...
extern static int illegal_a       = 42;
extern static const int illegal_b = 42;

} // namespace bar

Finally note that the extern examples in the snippet above are likely to result in -Wextern-initializer warnings although they are all perfectly legal. The warning is arguably intended to guard against multiple definitions of a variable name with external linkage, as the likely use case of extern, from the view of the compiler, is to declare or re-declare a given external linkage name in a header file (intended for several TU’s to use) or several source files, respectively, but to leave the definition of the variable to a single TU where extern can be omitted, as it has already been declared as such. If using this idiom (which is, along with the warning, sometimes controversial), extern sematically indicates not only that the name has external linkage but also that its definition is elsewhere, in the very source file where extern hs been omitted from the (re-)declaration and definition.

A recap of name lookup

We have briefly covered linkage, which governs whether a given scope may refer to given name or not. We’ll now proceed to name lookup, a mechanism which governs how the compiler may, when encountering a name, look for the declaration of that name. Lookup may seem to have a strong relation to the linkage of a name, but these two mechanisms are actually orthogonal; two same-name entities in two different TUs which both have internal linkage are strictly separate entities, not violating the One-Definition Rule (ODR), and name lookup for the respective name is concerned about the concept of scope rather than linkage. Linkage affects the scope for which a given name is referrable, but it has no direct relation to the name lookup procedure.

We’ll focus on name lookup for functions, as this is a case which contains the most quirks and also covers the context for which this post was written. For name lookup of a function name in a well-formed program, the result is at least one candidate function; if more than one candidate is found, a mechanism called overload resolution is sub-sequently used to pick the best candidate. There are quite a lot of rules and special cases for how overload resolution works, and we will not dwell any deeper into those in this particular post, but we’ll summarize the two main error cases w.r.t. function name lookup and overload resolution as:

  • name lookup finds no candidate function, or
  • name lookup finds several candidate functions but overload resolution cannot non-ambiguously select which of these candidate that is the best match.

Now, name lookup of a function will use one of the following procedures:

  • qualified name lookup,
  • unqualified name lookup,
  • argument-dependent lookup (ADL),

where the latter is only present for function names, thus our choice to focus on function name lookup.
